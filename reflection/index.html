What I Learned Building This

March 2025

The Cloud Resume Challenge sounded straightforward when I started: put a resume on Azure, add a visitor counter, wire up CI/CD. In practice, it turned into a crash course in all the things documentation doesn't prepare you for. Here's what actually stood out.

ARM templates are verbose but predictable

I went with a single ARM template instead of nested or linked ones. The JSON is long, but everything lives in one file, which made it much easier to debug failed deployments. When something breaks, you can trace the entire resource graph without jumping between files. The tradeoff is readability — ARM JSON is not pretty to look at — but for a project this size, the simplicity was worth it.

Azure Front Door replaced classic CDN (surprise)

My original plan was Azure CDN with the Standard_Microsoft SKU. Turns out Microsoft deprecated that, along with Standard_Verizon. The replacement is Azure Front Door Standard, which uses a completely different resource model — afdEndpoints, originGroups, and routes instead of the classic CDN endpoint structure. The documentation hadn't fully caught up with the deprecations when I was building this, so there was a stretch where I was piecing things together from API specs and forum posts. It works well now, but it wasn't a smooth migration path.

Python v2 functions need a hidden feature flag

Azure Functions for Python has a newer programming model (v2) that uses decorators instead of function.json files. It's much cleaner to write. But there's a catch: your function app won't discover any of your functions unless you set AzureWebJobsFeatureFlags=EnableWorkerIndexing in the app settings. This wasn't in the quickstart docs I followed. I spent a while staring at an empty functions list before finding the answer in a GitHub issue.

OIDC is worth the setup

GitHub Actions can authenticate to Azure using OpenID Connect federated credentials instead of storing client secrets in the repo. No secrets to rotate, no credentials that can leak. The initial setup takes more steps — you need an app registration, a service principal, and federated identity credentials for each repo and branch — but once it's configured, deployments just work. Given how many breaches start with leaked credentials, this felt like the right call.

DNS delegation keeps things clean

My main domain lives in Cloudflare, but Azure needs DNS control for things like domain validation and alias records. Rather than moving everything, I delegated just the cloudresume subdomain to Azure DNS by adding NS records in Cloudflare. The rest of my DNS stays untouched. It's a simple pattern that I'll probably reuse for future projects.

The real value

None of the individual pieces here are groundbreaking. But stringing them all together — static hosting, serverless functions, managed certificates, infrastructure as code, OIDC auth, CI/CD pipelines — forced me to deal with the integration problems that you only hit when you actually build something end to end. Reading docs is one thing. Debugging a CDN cache that won't purge at 11 PM is another.

If you're considering the Cloud Resume Challenge, I'd recommend it. Not because the resume itself matters, but because the project gives you a reason to touch every layer of the stack.
