What I Learned Building This

What a year of cloud building taught me

The Cloud Resume Challenge sounded simple enough. Put a resume on Azure, add a visitor counter, wire up CI/CD. What actually happened was a crash course in all the things documentation doesn't prepare you for.

One ARM template, one file, no regrets

I went with a single ARM template instead of nested or linked ones. The JSON is long, no question, but everything lives in one file and that made debugging failed deployments much easier. When something breaks you can trace the entire resource graph without jumping between files. ARM JSON is not pretty to look at, but for a project this size the simplicity won out.

So what happened to Azure CDN?

My original plan was Azure CDN with the Standard_Microsoft SKU. Microsoft deprecated that, along with Standard_Verizon. The replacement is Azure Front Door Standard, which uses a completely different resource model. You're working with afdEndpoints, originGroups, and routes instead of the classic CDN endpoint structure. The documentation hadn't caught up with the deprecations when I was building this, so I spent a stretch piecing things together from API specs and forum posts. It works well now, but the migration path was anything but smooth.

The feature flag nobody tells you about

Azure Functions for Python has a newer programming model (v2) that uses decorators instead of function.json files. Much cleaner to write. But your function app won't discover any of your functions unless you set AzureWebJobsFeatureFlags=EnableWorkerIndexing in the app settings. This was not in the quickstart docs I followed. I spent a while staring at an empty functions list before finding the answer buried in a GitHub issue.

OIDC over secrets, every time

GitHub Actions can authenticate to Azure using OpenID Connect federated credentials instead of storing client secrets in the repo. No secrets to rotate, no credentials that can leak. The initial setup takes more steps because you need an app registration, a service principal, and federated identity credentials for each repo and branch. Once it's configured though, deployments just work. Given how many breaches start with leaked credentials, this felt like the obvious choice.

Delegating DNS without losing control

My main domain lives in Cloudflare, but Azure needs DNS control for things like domain validation and alias records. Rather than moving everything, I delegated just the cloudresume subdomain to Azure DNS by adding NS records in Cloudflare. The rest of my DNS stays untouched. Simple pattern, and one I'll probably reuse for future projects.

What I actually learned

None of the individual pieces here are groundbreaking. But stringing them all together forced me to deal with integration problems that you only hit when you actually build something end to end. Static hosting, serverless functions, managed certificates, infrastructure as code, OIDC auth, CI/CD pipelines. Reading docs is one thing. Debugging a CDN cache that won't purge at 11 PM is another.

If you're considering the Cloud Resume Challenge, do it. Not because the resume itself matters, but because the project gives you a reason to touch every layer of the stack.
